Multi-vault strategy

Separate the worlds for yourself. Let the AI work across them for you.

The live Mycelium orientation does not reveal a single notebook. It reveals a configured ecosystem of vaults. That is the strategic idea: you keep cognitive, brand, legal, client, archive, product, and memory boundaries intact, while the AI gets tools that can cross those boundaries deliberately.

For the human: clean separation

Large domains can stay in their own vaults: a platform vault, a client-project vault, a brand vault, a product vault, an archive, a memory vault, or a legal/process vault. You do not have to collapse everything into one enormous vault just so an AI can find it.

For the AI: one operating surface

Mycelium gives the AI cross-vault tools: search across every vault, find a note when only the title is known, trace links between vaults, get ecosystem statistics, and optionally search all vaults by meaning.

What we can do together

Useful work, not just access.

The toolset is broad, but the user benefit is simple: the AI can find, read, organize, write, query, and maintain your own notes across separate vaults without turning your knowledge into someone else's cloud.

Find the right notes

Search by text, by meaning, by tag, by file name, by folder, by frontmatter property, or across every configured vault at once.

Read with context

Open whole files, inspect outlines, follow wikilinks, gather backlinks, and pull enough surrounding context to understand why a note matters.

Edit without clumsiness

Create notes, update frontmatter, append to sections, move files, rename notes, and keep wikilinks pointing to the right place.

Query like a database

Use properties, tags, Bases, and Dataview-like filters to turn notes in separate domains into lists, dashboards, and decision surfaces.

Maintain the vault

Spot broken links, unresolved links, dead ends, orphan notes, stale notes, duplicate paths, and other quiet sources of entropy.

Stay local-first

The server runs on your machine. Your vaults remain markdown on disk. Optional embeddings can run through local Ollama.

Advanced operator layer

When Obsidian is running, the AI can work inside the living app.

The filesystem tier gives Mycelium sovereignty and reliability: markdown on disk, available even when Obsidian is closed. The CLI tier adds something different. It lets an experienced AI and Obsidian operator reach into the running application: commands, Bases, templates, sync, history, plugins, snippets, themes, hotkeys, recents, and runtime APIs.

The CLI tier is the app-aware layer

This is where Mycelium stops being only a file operator and starts becoming an Obsidian operator. The AI can use Obsidian's own search, create from your templates, inspect Bases views, compare and restore history, read sync state, and run commands from the command palette.

Use Obsidian structuresvault_search query_base list_base_views

Recover and comparediff_versions restore_version sync_history

eval_obsidian is the power hatch

eval_obsidian runs JavaScript inside Obsidian with access to the app runtime. That means the AI can inspect internal state, use Obsidian APIs, reach capabilities that are not yet wrapped as formal tools, and prototype highly specific operations.

This is the advanced world. It belongs with experienced AI and Obsidian operators who understand the vault, the app, and the consequences of runtime access.

Deep inspectionAsk the live app what it knows, not only what files contain.
Plugin bridgesExplore APIs exposed by installed plugins.
Custom operationsDo precise runtime work before it deserves a dedicated tool.
Operator choiceDisable it with OBSIDIAN_DISABLED_TOOLS when a deployment should stay conservative.
Ownership and responsibility: this power is not a scare warning; it is the honest shape of local ownership. The human operator owns the vault, chooses which tools are exposed, decides when the app-aware layer is appropriate, and can disable advanced tools for safer everyday deployments.

User-friendly workflows

What the tools add up to.

These examples are written for a human deciding what Mycelium is for. The exact tool names stay available as proof underneath the plain language.

Start a session inside a large vault

Orient before acting. The AI can map the configured vaults, see which tools are available, and choose the safest route.

1Run orientation and discover the available surface.
2Check vault names, tool tiers, and disabled tools.
3Search, read, or query with the right retrieval mode.

Turn a scattered topic into a cited synthesis

The AI can search across vaults, follow promising notes, read a small subset deeply, and produce an answer tied back to source files.

1Search all vaults for the topic and related phrases.
2Find dedicated notes, backlinks, and local context.
3Read only the notes that matter and synthesize.

Ask across domains without merging them

A product question may touch brand, platform, client work, development notes, old strategy, and memory. Mycelium lets the AI cross those boundaries while the vaults stay separate for you.

1Start with get_started to see the configured vault names.
2Use search_all_vaults or find_note_by_name when the location is uncertain.
3Trace cross-vault links and cite which vault each piece of evidence came from.

Clean up a vault without losing its shape

Use health reports, broken-link checks, and stale-note scans to find the places where one vault or the whole ecosystem is fraying.

1Run a vault health report.
2Review broken links, orphans, dead ends, and stale notes.
3Move, rename, merge, or repair with wikilinks updated.

Build a lightweight operating system in markdown

Properties, tasks, tags, Bases, daily notes, and templates let each vault behave like its own working environment while still belonging to the larger surface.

1Use frontmatter properties and tags as the organizing layer.
2Query notes by status, type, folder, or priority.
3Create from templates and update daily notes or task lists.

Setup and limits

Clear boundaries make the product more trustworthy.

Always available

The filesystem tier works with plain Node.js and markdown files. Obsidian does not need to be open for these 74 tools.

App-powered tools

The 28 CLI tools need Obsidian 1.12+ running with the command line interface enabled. If the app is closed, they fail cleanly.

Optional semantic search

Meaning-based search requires local Ollama and an embedding index. Everything else keeps working without it.

Power tools need care

update_file replaces the file body. update_section on H1 reaches the end of a file. search_replace_in_file should be used deliberately.

Tools can be disabled

Operators can remove risky or unwanted tools from the exposed MCP surface with OBSIDIAN_DISABLED_TOOLS.

Local-first by design

The server is software you run. The vault remains yours: local files, local configuration, and optional local embeddings.
